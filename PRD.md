# PRD: CXM 워크플로우 자동화 시스템

> **대상 워크플로우 단계**: `고객 문의` → `PM 문제 파악` → `엔지니어 전달` → `고객 안내` → `엔지니어 문제해결` → `QA` → `고객 안내`

---

## 1. 배경 및 목적

### 회사 컨텍스트
- **업종**: B2B AI API 공급사
- **고객 규모**: 약 150개 유료 고객사 (맞춤형 AI API 제공)
- **비즈니스 모델**: PoC(약 3개월) → 맞춤형 솔루션 구현 → 1년 구독 전환
- **CXM 인원**: 3명

### 문제 정의

#### 문제 1. CS 대응 워크플로우가 규격화되지 않음
- CS 유형은 크게 두 가지: **(1) 단순 에러 대응**, **(2) 추가 기능 요구 및 문의**
- 3명의 CXM이 매 문의마다 고객사 맥락을 처음부터 파악해야 함
  - 제공 중인 서비스 확인 → 요청사항 파악 → 기존 소통 맥락 파악
  - **노션, 채널톡, Slack** 등 3~4개 툴을 순차적으로 확인해야 하는 구조
- 결과: 맥락 파악에만 **15~30분 소요** → 실제 대응 지연

#### 문제 2. 구독 유지를 위한 선제적 제안 부재
- PoC 종료 후 추가 개발/제안 없이 방치
- 기술 발전으로 고객사가 자체 대체 솔루션 도입 후 구독 해지 (예: background removal 서비스 → nanobanana/Gemini로 대체)
- lock-in 장치 부재: 신기술 기반 신규 서비스 제안, 경쟁 서비스 한계 보완 등

## 2. 목표 (Goals)

| # | 목표 | 개선 대상 워크플로우 단계 |
|---|------|--------------------------|
| G1 | CS 문의 접수 시 고객사 맥락을 2분 이내에 파악 | `고객 문의` → `PM 문제 파악` |
| G2 | 맥락 확인 소스를 1개 대시보드로 통합 | `PM 문제 파악` → `엔지니어 전달` |
| G3 | 이탈 위험 고객사를 사전에 감지하여 선제적 제안 | `고객 안내` 이후 관계 유지 단계 |

### 기대 KPI

| 지표 | AS-IS | TO-BE |
|------|-------|-------|
| 맥락 파악 소요 시간 | 15~30분 | 2분 이내 (90%↓) |
| 확인해야 할 소스 수 | 3~4개 툴 | 1개 대시보드 (75%↓) |
| 선제적 제안 건수 | 월 0건 | 월 5건+ |
| 이탈 위험 감지 | 사후 대응 | 사전 알림 |

---

## 3. 범위 (Scope)

### In-Scope
- 고객사 맥락 자동 수집 및 요약 (노션 + 채널톡 + Slack 통합)
- 고객사별 현황 대시보드 (서비스 현황, CS 이력, 액션 아이템)
- 이탈 위험 신호 감지 및 알림

### Out-of-Scope (v1)
- 엔지니어 작업 티켓 자동 생성
- 고객사 직접 노출 포털
- 자동 응답 챗봇

---

## 4. 기능 요구사항

### Feature 1. 고객사 맥락 자동 수집/요약 `[고객 문의 → PM 문제 파악]`
| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F1-1 | CS 문의 접수 시 해당 고객사 정보를 노션/채널톡/Slack에서 자동 수집 | P0 |
| F1-2 | 수집된 정보를 1페이지 요약본으로 자동 생성 (제공 서비스, 최근 CS 이력, 담당자) | P0 |
| F1-3 | 요약본을 Slack 또는 대시보드에 자동 전달 | P1 |

### Feature 2. 고객사 대시보드 `[PM 문제 파악 → 엔지니어 전달]`
| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F2-1 | 150개 고객사 목록 및 서비스 현황 한눈에 보기 | P0 |
| F2-2 | 고객사별 CS 이력 및 진행 중인 액션 아이템 표시 | P0 |
| F2-3 | CXM 담당자 배정 및 상태 업데이트 기능 | P1 |

### Feature 3. 이탈 위험 감지 및 선제적 제안 `[관계 유지]`
| ID | 요구사항 | 우선순위 |
|----|----------|----------|
| F3-1 | 이탈 위험 신호 정의 및 자동 감지 (CS 빈도 급증, 문의 유형 변화, 갱신일 임박 등) | P1 |
| F3-2 | 위험 고객사에 대한 알림 발송 (담당 CXM에게) | P1 |
| F3-3 | 신기술/경쟁사 동향 기반 제안 초안 자동 생성 | P2 |

## 5. 기술 스택 (안)

| 영역 | 후보 |
|------|------|
| 데이터 소스 | Notion API, 채널톡 API, Slack API |
| 자동화 엔진 | Claude API (요약/분석), n8n 또는 Make (워크플로우) |
| 대시보드 | Notion 기반 또는 경량 웹앱 (Next.js/Streamlit) |
| 알림 | Slack 웹훅 |

---

## 6. 성공 측정 기준

- [ ] 맥락 파악 시간 2분 이내 달성률 90% 이상 (2주 운영 후 측정)
- [ ] CXM 전원 대시보드 단일 소스로 전환 완료
- [ ] 월 5건 이상 선제적 제안 발송
- [ ] 이탈 위험 감지 알림 → 실제 대응 리드타임 단축 확인

---

## 7. 개발 우선순위 (Phased Rollout)

| Phase | 범위 | 목표 |
|-------|------|------|
| Phase 1 | F1-1, F1-2, F2-1, F2-2 | 맥락 파악 시간 90% 단축, 소스 통합 |
| Phase 2 | F1-3, F2-3, F3-1, F3-2 | 실시간 알림 + 이탈 감지 |
| Phase 3 | F3-3 | 선제적 제안 자동화 |
